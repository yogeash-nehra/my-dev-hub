## [BREAKING] Breaking Changes

### [BREAKING] agent-memory-2026-07-22 Behavioral Change Takes Effect Today

**Source:** Anthropic Platform Release Notes | **Date:** July 22, 2026 (announced July 2, 2026) | **Link:** https://platform.claude.com/docs/en/release-notes/overview

**What changed:** The `managed-agents-2026-04-01` beta header, when used on `GET /v1/memory_stores/{memory_store_id}/memories`, now behaves identically to `agent-memory-2026-07-22`. Results are returned in a stable server-defined order and the `order_by`/`order` query parameters are silently ignored; `depth` only accepts `0`, `1`, or omitted (other values now return `400`); and `path_prefix` must end with `/` and matches whole path segments instead of substrings. Page cursors issued before today are invalid under the new behavior — restart pagination from the first page.

**TL;DR:** As of today, any call to `GET /v1/memory_stores/{id}/memories` with `managed-agents-2026-04-01` silently ignores `order_by`/`order` and returns 400 on non-standard `depth` values — audit and update memory store queries now.

**Developer signal:** If you use the Anthropic SDK at current versions (Python ≥0.116.0, TypeScript ≥0.110.0, Go ≥1.56.0, Java ≥2.48.0, Ruby ≥1.55.0, PHP ≥0.36.0, C# ≥12.35.0, CLI ≥1.16.0), the SDK already sends `agent-memory-2026-07-22` on all memory store calls — you opted in three weeks ago and this is a non-event. The risk is in code that manually sets the `betas` array on memory store calls and includes `managed-agents-2026-04-01`. That code now silently adopts new behavior. Audit checklist: (1) search your codebase for any calls to `/v1/memory_stores/` that pass explicit beta headers; (2) remove any `order_by`, `order`, or non-standard `depth` parameters — the API no longer honors them and will 400 on invalid depth; (3) restart pagination — all cursors from before today are invalid with the new behavior; (4) confirm you are not sending both headers on the same request (already a 400, still a 400). Session endpoints — not memory store endpoints — continue to use `managed-agents-2026-04-01` normally; don't change those.

**Affects you if:** You call `GET /v1/memory_stores/{memory_store_id}/memories` and explicitly set the `betas` parameter to include `managed-agents-2026-04-01`, especially with `order_by`, `order`, non-standard `depth` values (e.g. `depth=5`), or `path_prefix` without a trailing slash.

**Adoption effort:** Quick — audit and remove deprecated query parameters; restart pagination; if you manually set beta headers on memory store calls, replace `managed-agents-2026-04-01` with `agent-memory-2026-07-22` there.

**Primary source:** https://platform.claude.com/docs/en/release-notes/overview (July 2 entry details the behavior; July 22 is the enforcement date for `managed-agents-2026-04-01`)

## API & SDK Changes

### [MEDIUM] Claude Managed Agents: Effort Control, Session Seeding, Lifecycle Webhooks, Thread Streaming, Version-Optional Updates

**Source:** Anthropic Platform Release Notes | **Date:** July 22, 2026 | **Link:** https://platform.claude.com/docs/en/release-notes/overview

**What changed:** Five coordinated additions to Claude Managed Agents landed today: (1) `effort` level on the agent model configuration object at creation time; (2) webhooks covering `environment.*` (four types) and `memory_store.*` (three types) lifecycle events; (3) session seeding with `initial_events` — up to 50 `user.message` and `user.define_outcome` events at `POST /v1/sessions`, starting the agent loop in the same call; (4) thread-level event stream deltas on `GET /v1/sessions/{session_id}/threads/{thread_id}/stream` via the `event_deltas[]` query parameter; (5) `version` field now optional on agent updates. Anthropic SDK Python v0.118.0 ships all five.

**TL;DR:** Claude Managed Agents now supports per-agent effort-level control, environment and memory store lifecycle webhooks, session pre-seeding with up to 50 events at creation (agent loop starts immediately), per-thread delta streaming of subagent text, and unconditional agent updates — all live today, supported in Python SDK v0.118.0.

**Developer signal:** Five changes, each targeting a distinct integration gap:

**(1) Effort on model config** — Pass `effort` inside the `model` object when you create an agent (`POST /v1/agents`). This sets a default effort level for that agent's sessions. Previously, effort control required per-call configuration. Now you can define separate agents for high-quality vs. throughput-focused workloads and let the agent definition carry the effort setting. See [Effort levels](https://platform.claude.com/docs/en/build-with-claude/effort#effort-levels).

**(2) Environment and memory_store webhooks** — Four `environment.*` event types and three `memory_store.*` event types now available in the webhooks subscription. Previously, knowing when a sandbox environment was ready or a memory store updated required polling the status endpoint. Subscribe via [Subscribe to webhooks](https://platform.claude.com/docs/en/managed-agents/webhooks#supported-event-types). This closes a polling gap in event-driven agent orchestration.

**(3) Session seeding with `initial_events`** — Pass `initial_events` (array of `user.message` and/or `user.define_outcome` events, up to 50) on `POST /v1/sessions`. If the list is non-empty, the agent loop starts immediately in the same call — no separate send-events request needed to kick off work. This reduces round-trips for initialization-heavy agents: you can deliver user context, task state, and an outcome definition in a single API call and receive the session ID with the agent already running. See [Seed the session with initial events](https://platform.claude.com/docs/en/managed-agents/sessions#seed-the-session-with-initial-events).

**(4) Thread stream event deltas** — `GET /v1/sessions/{session_id}/threads/{thread_id}/stream` now accepts `event_deltas[]` with the same options as the session-level stream. Preview a subagent's text as it generates, before the complete `agent.message` event arrives. **Migration note:** the delta type is `content_delta`, not `content_block_delta` — accumulator code written for the Messages API streaming does **not** carry over unchanged. Rewite any accumulator that handles these events. See [Preview session thread events](https://platform.claude.com/docs/en/managed-agents/events-and-streaming#preview-session-thread-events).

**(5) Version field optional on agent update** — Supply `version` to `PATCH /v1/agents/{id}` for optimistic concurrency (409 on mismatch), or omit it to apply the update unconditionally. Previously required; now opt-in. See [Update semantics](https://platform.claude.com/docs/en/managed-agents/agent-setup#update-semantics).

SDK: `pip install --upgrade anthropic` (v0.118.0) for all five additions.

**Affects you if:** You build on Claude Managed Agents, specifically: if you want per-agent cost/quality differentiation (1); if you poll for environment or memory lifecycle state (2); if your agent sessions require multi-turn context initialization (3); if you want real-time subagent text streaming (4); if you do concurrent agent updates and want simpler unconditional writes (5).

**Adoption effort:** Moderate — each addition requires targeted integration work. Items 1, 3, and 4 are the most impactful for active builders.

**Primary source:** https://platform.claude.com/docs/en/release-notes/overview

### [MEDIUM] LangChain Standardizes `reasoning_effort` Across Five Provider Integrations

**Source:** LangChain GitHub Releases | **Date:** July 21, 2026 | **Link:** https://github.com/langchain-ai/langchain/releases

**What changed:** `langchain-core==1.5.0` introduces `reasoning_effort` as a standard chat model parameter, and five provider integrations released simultaneously: `langchain-anthropic==1.5.0`, `langchain-openai==1.4.0`, `langchain-xai==1.3.0`, `langchain-fireworks==1.5.0`. Previously, controlling thinking effort required provider-specific kwargs or custom wrappers. `langchain-anthropic==1.5.0` also adds an `advisor_` prefix for automatic recognition of Anthropic built-in tools (web search, code execution, etc.), reducing the need to manually define their schemas.

**TL;DR:** LangChain makes `reasoning_effort` a portable standard parameter across Anthropic, OpenAI, xAI, and Fireworks integrations as of July 21 — set it at model creation or per-call; critical caveat: the interface is standardized but the underlying behavior is not.

**Developer signal:** Four things to know: (1) If you currently set thinking-related options via `model_kwargs` or provider-specific interfaces, migrate to `reasoning_effort="low"|"medium"|"high"` at model initialization or per-invocation. Simpler and more portable. (2) The critical caveat: LangChain standardized the *interface*, not the *semantics*. `reasoning_effort="high"` on Anthropic maps to extended thinking at Anthropic's default high budget; on OpenAI it affects o-series chain-of-thought depth; on xAI and Fireworks it depends on the model and its reasoning implementation. Do not assume behavioral equivalence — test each backend independently for latency, cost, and output quality at the same parameter string. (3) If you use `langchain-anthropic`, the new `advisor_` prefix means LangChain now auto-recognizes Anthropic built-in tools by name prefix — reduces manual schema boilerplate when using web search or code execution tools. (4) Update all five packages together; `langchain-core==1.5.0` is the dependency for the standard parameter — provider packages depend on it.

**Affects you if:** You use LangChain with Anthropic, OpenAI, xAI, or Fireworks backends and want portable effort control; you use `langchain-anthropic` with Anthropic built-in tools and want to reduce boilerplate.

**Adoption effort:** Quick — update package versions and optionally migrate from provider-specific kwargs to `reasoning_effort`. Not a breaking change; old kwargs still work.

**Primary source:** https://github.com/langchain-ai/langchain/releases

## Trends & Emerging Tech

### Reasoning Effort Becomes a Standard Control Surface — But the Interface Hides the Implementation Gap

**Source:** LangChain GitHub Releases | **Date:** July 21, 2026 | **Link:** https://github.com/langchain-ai/langchain/releases

**What's happening:** LangChain's simultaneous release of `reasoning_effort` across five provider integrations signals that the ML framework ecosystem is treating thinking-effort control the same way it already treats `temperature` — as a universal dial developers expect to set regardless of backend. This follows Anthropic's own `effort` parameter in the Messages API and OpenAI's `reasoning_effort` on o-series. The standardization happened fast: in under a year, "how much should this model think?" went from a niche, model-specific option to a parameter frameworks standardize as a matter of course.

**Why watch this:** The gap between the interface and the implementation matters more than the standardization itself. A multi-provider LangChain application using `reasoning_effort="high"` will produce code that is superficially portable but behaviorally inconsistent — Anthropic's "high" triggers extended thinking with Anthropic's budget logic; OpenAI's affects o-series in a different way; xAI and Fireworks have their own mappings. Teams doing cost benchmarks or A/B comparisons across providers at the same parameter string will see unexplained variance until they test each backend independently. Watch how the next capabilities to get standardized (vision, audio, structured output) handle the same tension — portable interface vs. fragmented semantics is now the default state of the multi-provider LLM framework layer.

**[TENSION] Portable interface, fragmented semantics: `reasoning_effort` standardization creates a false equivalence floor**
LangChain standardizing `reasoning_effort` across providers (this digest, Tooling) creates a single parameter over at least four distinct implementation architectures. Anthropic's effort parameter maps to extended thinking token budgets; OpenAI's reasoning_effort affects o-series chain-of-thought depth; xAI and Fireworks implement it differently again. A developer writing `reasoning_effort="high"` in a multi-provider pipeline will produce code that looks identical across backends but delivers different quality, latency, and cost profiles on each. The same pattern has played out with `temperature` (standard parameter, non-equivalent behavior across deterministic vs. stochastic models) and `max_tokens` (standard name, non-equivalent truncation semantics). The tension: frameworks win on portability, but the uniformity of the interface actively masks variance that engineers need to benchmark independently. Teams building production multi-provider systems should treat each provider as requiring its own effort calibration run, regardless of the shared parameter name.
*Grounded in: LangChain reasoning_effort standardization (this digest, Tooling); reasoning_effort implementation notes from Anthropic effort docs and OpenAI o-series docs.*

**[BUILDER'S ANGLE] `initial_events` + thread delta streaming = single-round-trip agent bootstrapping with live output**
Two of today's Managed Agents additions combine into a pattern that wasn't practical before. Previously: initialize a session, send a separate message to start work, then poll or subscribe to the session stream to see output — three interactions minimum before seeing anything. Now: `POST /v1/sessions` with `initial_events` containing user context + `user.define_outcome` (agent loop starts immediately in the same call), then subscribe to `GET /v1/sessions/{id}/threads/{thread_id}/stream` with `event_deltas[]` to see subagent text streaming in real time. A front-end showing a user "your agent is working on X" can go from session creation to live streaming output in two API calls with no intermediate polling. The practical shape: customer-facing agent products with rich initialization context (user profile, session history, task state) now have a cleaner initialization path. The constraint to watch: `initial_events` is capped at 50 events, and thread streaming accumulator code differs from the Messages API (use `content_delta`, not `content_block_delta`).
*Grounded in: Claude Managed Agents `initial_events` and thread `event_deltas` (this digest, API & SDK Changes); Anthropic SDK Python v0.118.0 (this digest, Quick Hits).*

**[OPEN QUESTION] Who can subscribe to a thread stream in a multi-agent session, and does the answer match what operators expect?**
Thread-level `event_deltas` lets a client subscribe to a specific subagent's thread and preview its text as it generates. That's useful for debugging and UX — but it raises an access question that the release notes don't explicitly address: in a session that spawns multiple subagents, each working on different parts of a task, who is authorized to subscribe to which thread? The session owner has session-level stream access, but thread-level access may expose subagent reasoning or intermediate results that the operator intended to be internal. If you build a product where end users interact with a parent agent but don't need visibility into subagent work, you'll need to be deliberate about which thread streams you expose through your own API layer. This is the same kind of implicit permission gap that the OpenAI Erdős model incident surfaced in a different context (July 21 digest): as long-horizon agent sessions gain more observability surface, the authorization model for that surface needs to be designed explicitly, not inherited from the session-level defaults.

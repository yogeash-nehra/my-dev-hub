## API & SDK Changes

### [HIGH] LiteLLM CVE-2026-49468 — Critical Auth Bypass via Host Header Injection
**Source:** GitLab Advisory DB / cybersecuritynews.com | **Date:** June 16–17, 2026 | **Link:** https://advisories.gitlab.com/pypi/litellm/CVE-2026-49468/
**What changed:** A critical authentication bypass vulnerability was disclosed in LiteLLM's proxy layer, affecting all deployments running a version before 1.84.0.
**TL;DR:** An unauthenticated attacker can bypass LiteLLM proxy auth by injecting a crafted `Host` header — CVSS 9.8 Critical (AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:H), patched in v1.84.0 (current stable: v1.89.2).
**Developer signal:** If you're running `litellm --proxy` on any version older than 1.84.0, this is an active critical exposure. The flaw is in `get_request_route()` in `litellm/proxy/auth/auth_utils.py`: the function uses `request.url.path` to determine the effective request path, but Starlette reconstructs this path using the client-supplied `Host` header, allowing an attacker to craft requests that bypass authentication entirely with no credentials. The current stable is 1.89.2; there is no configuration-level workaround that fully mitigates this — upgrade is the only fix. LiteLLM Cloud customers are not affected. A previous LiteLLM CVE from April 2026 (CVE-2026-42208, SQL injection) was actively exploited in the wild within 36 hours of disclosure — treat this disclosure with the same urgency.
**Affects you if:** You are running the open-source LiteLLM proxy server on a version before v1.84.0, especially if it is exposed to any network (internal or external).
**Adoption effort:** Quick (version upgrade only; no code or config changes required beyond `pip install litellm==1.89.2`).
**Primary source:** https://advisories.gitlab.com/pypi/litellm/CVE-2026-49468/ | GitHub GHSA: GHSA-4xpc-pv4p-pm3w

### MCP Is Converging as the Default Agent Tool Protocol Across Managed Platforms
**Source:** AWS AgentCore Gateway | **Date:** June 17, 2026 | **Link:** https://aws.amazon.com/blogs/aws/announcing-web-search-on-amazon-bedrock-agentcore-ground-your-ai-agents-in-current-accurate-web-knowledge/
**What's happening:** Both AWS and Google have now shipped MCP-native tooling as the default integration surface for their managed agent platforms: Web Search on Bedrock AgentCore connects via AgentCore Gateway using MCP as the connector protocol (GA, June 17), and Google's Managed Agents in the Gemini API launched at I/O with MCP as the agent-to-tool protocol. Neither platform requires custom tool-wiring code — tools are configured at the Gateway or agent-harness level using MCP connector definitions. Anthropic's Claude Managed Agents also uses MCP for tool routing via the `managed-agents-2026-04-01` interface.
**Why watch this:** With all three major managed-agent platforms (AWS, Google, Anthropic) now shipping MCP-native connectors, a tool server written once as an MCP server is likely to work across all three platforms without modification. The practical implication: if you're building custom tools for agents, implementing them as MCP servers now rather than proprietary SDK integrations gives you maximum portability. Expect the set of built-in managed MCP targets (web search, document retrieval, code execution, calendar, CRM) to expand rapidly on all three platforms over the next quarter — the pattern is set.

**[OPEN QUESTION] Does Bedrock Managed Knowledge Base displace the vector DB middleware market, or only capture the AWS-captive workload?**
Amazon Bedrock Managed Knowledge Base (GA, June 17) takes the full RAG stack in-house for Bedrock-based agents — ingestion, chunking, embedding, storage, retrieval — removing the need to operate a vector database. Google's Vertex AI Vector Search and Anthropic's Claude Managed Agents memory (GA April 2026) have made similar moves on their own platforms. But for developers who start new agent projects today on a single cloud, the default path now bypasses the vector DB entirely. The bottom of the market — developers building net-new on AWS, GCP, or Azure — may shift toward managed options over the next year; the enterprise multi-cloud tier is likely safe.

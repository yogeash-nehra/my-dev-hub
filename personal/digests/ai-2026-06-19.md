**What changed:** Codex gains a macOS-native Record & Replay capability: users demonstrate a workflow once on their Mac, Codex captures the full sequence and converts it into a reusable "skill" that can be triggered repeatedly without re-demonstration, and shared across a team.
**TL;DR:** OpenAI Codex now lets you record a workflow on macOS and replay it as an automated, shareable skill — the first demo-to-skill recording capability shipped by a major AI coding tool, alongside Browser Use improvements and bulk automation run history actions.
**Developer signal:** Record & Replay is meaningfully different from Codex's existing agentic coding: instead of prompting the model to figure out a task from scratch each time, you demonstrate it once (filing expenses, publishing a release, booking a resource) and Codex stores that demonstrated path as a reusable skill. The critical open question is how robust the recording is to UI changes — traditional RPA tools (UiPath, Automation Anywhere) record exact coordinates and break when the UI shifts; if Codex uses model understanding of intent to make skills more resilient, this is a fundamentally different capability. Skills are shareable: one employee's recording becomes the team's automation, reducing drift between "what the agent does" and "what a human actually does." The June 18 update also includes: bulk actions in automation run history, new deep links for SSH connection management, and Browser Use improvements where visible-tab routing and annotations now persist when a draft browser session moves to server-side execution. Availability caveats: macOS only; explicitly excluded from EEA, UK, and Switzerland at launch; Computer Use must be enabled on your account or by your org admin. If you're in a supported region with Computer Use enabled, this is worth testing now for repetitive GUI workflows that are currently handled manually.
**Affects you if:** You use OpenAI Codex on macOS for agentic tasks with Computer Use enabled; or you are building team automation workflows with Codex and want repeatable, shareable execution paths.
**Adoption effort:** Quick (built into the macOS Codex app update; no API changes required).

**Source:** AWS Summit NYC / strands-agents/evals GitHub | **Date:** June 17–18, 2026 | **Link:** https://aws.amazon.com/blogs/aws/top-announcements-of-the-aws-summit-in-new-york-2026/

**TL;DR:** AWS Strands Agents ships three production-hardening capabilities at Summit NYC: sub-ms sandbox launch (Shell), chaos testing + red teaming at build time (Evals v1.0 with five built-in attack strategies), and a context management layer that halves costs — 50M downloads since May open-source launch.
**Developer signal:** Strands Evals v1.0.0 is the piece with the most immediate developer value — add it to your CI pipeline now. ### [NOTABLE] llama.cpp b9723: Eagle3 Speculative Decoding Now Supports Qwen 3.5 and 3.6
**Source:** ggml-org/llama.cpp | **Date:** June 19, 2026 | **Link:** https://github.com/ggml-org/llama.cpp/releases/tag/b9723
**What changed:** llama.cpp adds Eagle3 speculative decoding support for Qwen 3.5 and 3.6 model families, extending the Eagle3 draft framework (previously covering select model families) to Alibaba's latest Qwen generation, plus deferred boundary checkpoint restore for hybrid model architectures.
**TL;DR:** Qwen 3.5 and 3.6 users on llama.cpp can now use Eagle3 speculative decoding — Eagle-family methods typically deliver 2-4x inference speedup at equivalent output quality vs. standard autoregressive generation.
**Developer signal:** Eagle3 is the third generation of the Eagle speculative decoding approach, using a lightweight draft model to predict multiple tokens ahead, then verifying them in parallel with the target model. If you're running Qwen 3.5 or 3.6 locally via llama.cpp and have (or can download) the matching Eagle3 draft model weights, update to b9723 and add `--draft-model <path-to-eagle3-draft>` to your `llama-server` or `llama-cli` invocation. The "deferred boundary checkpoint restore" change handles Qwen's hybrid MoE-dense layer architecture — where MoE layers and dense layers alternate — ensuring correct state restoration at layer-type boundaries, which is a correctness fix that also enables the Eagle3 draft to work reliably across the full model. Eagle3 draft model weights for Qwen 3.5/3.6 should be available on Hugging Face; search for `Eagle3-Qwen3` variants. No other configuration changes are required beyond the standard speculative decoding setup.
**Affects you if:** You are running Qwen 3.5 or Qwen 3.6 models locally via llama.cpp and want significantly faster inference without hardware upgrades.
**Adoption effort:** Moderate (requires downloading the Eagle3 draft model weights for your Qwen variant and adding `--draft-model` to your invocation; no other code changes).
**Primary source:** https://github.com/ggml-org/llama.cpp/releases/tag/b9723

### AI Developer Tooling Pivots from Capability to Reliability
**Source:** OpenAI Developers + AWS Summit NYC | **Date:** June 17–19, 2026 | **Link:** https://developers.openai.com/codex/record-and-replay
**What's happening:** Three separate capabilities shipped in roughly one 24-hour window — Codex Record & Replay, Strands Shell, and Strands Evals v1.0 — each focused on making AI agent behavior reliable and reproducible rather than more capable. Record & Replay gives you deterministic workflow replay. Strands Evals gives you adversarial pre-deployment testing with built-in red team attack strategies. Strands Shell gives you the fast isolated execution environment that makes chaos testing in CI economically viable. None of these make models smarter; all three make AI systems more trustworthy in production.
**Why watch this:** This may signal a maturation phase in AI dev tooling — where the key adoption bottleneck isn't raw model capability but reliability, testability, and workflow fidelity. As Strands Evals and similar frameworks normalize adversarial testing for agents, expect "passed evals?" to become a standard CI gate for agent deployments the way "passed unit tests?" already is for traditional code. Watch for LangChain, LlamaIndex, AutoGen, and CrewAI to ship comparable pre-deployment testing features in the next quarter — once one major agent SDK ships it, the rest tend to follow within 60-90 days.

**[OPEN QUESTION] When does adversarial agent testing become a standard CI gate — and what does "passing" mean?**
Strands Evals v1.0 (June 16-18) ships chaos testing (deterministic fault injection: tool timeouts, network errors, response corruption) and adversarial red teaming (Crescendo, GOAT, PAIR, BadLikertJudge, SequentialBreak attack strategies) as pre-deployment evaluation tools with CI integration via Strands Shell's sub-ms sandbox. The infrastructure is now there. The open question: what does "passing" mean for an agent? Unit tests have binary pass/fail criteria; agent evals don't, because outputs are non-deterministic and quality is context-dependent. Strands Evals sidesteps this partially by focusing on behavioral properties under adversarial pressure (does the agent refuse correctly under PAIR attacks? does it degrade gracefully under tool timeouts?) rather than output quality scoring — but that covers only a subset of what "reliable agent" means. The harder research problem is defining agreed-upon behavioral benchmarks for agents that organizations can adopt as shared CI gates, equivalent to how "code coverage >80%" is a conventional (if imperfect) bar for test suites. Strands Evals v1.0 is the best available starting point; it's not the end state.
*Grounded in: Strands Evals v1.0.0 chaos testing and red team features (this digest, Tooling); built-in attack strategies (Crescendo, GOAT, PAIR, BadLikertJudge, SequentialBreak) documented at github.com/strands-agents/evals; Strands Shell sub-ms launch enabling CI use (this digest)*

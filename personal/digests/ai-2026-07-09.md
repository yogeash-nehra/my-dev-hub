### [BREAKING] Gemini preview model strings sunset — three endpoints go dark today (July 9, 2026)

**Source:** Google AI for Developers | **Date:** July 9, 2026 | **Link:** https://ai.google.dev/gemini-api/docs/deprecations

**What changed:** Three Gemini preview model identifiers reach end-of-life and stop accepting requests on July 9, 2026: `gemini-3.1-flash-lite-preview`, `gemini-2.5-flash-lite-preview-09-2025`, and `gemini-2.5-flash-preview-09-2025`. Calls to any of these strings now return errors.

**TL;DR:** Three Gemini preview model strings are disabled as of today; migrate to `gemini-3.1-flash-lite` (cost-efficient stable) or `gemini-3.1-flash` (higher quality stable) to restore service.

**Developer signal:** Scan your codebase, configs, and environment variables for any of the three deprecated strings — they appear in constants, YAML config, `.env` files, and third-party proxy configs (LiteLLM, LangChain, etc.). Migration paths: (1) `gemini-3.1-flash-lite-preview` → `gemini-3.1-flash-lite` (direct stable replacement, same capability tier); (2) `gemini-2.5-flash-lite-preview-09-2025` → `gemini-3.1-flash-lite` (most cost-efficient equivalent) or `gemini-3.1-flash` for better quality; (3) `gemini-2.5-flash-preview-09-2025` → `gemini-3.1-flash` stable. If you're hitting this via a managed provider or API gateway, check whether they've updated their model routing — some proxy layers hardcode model strings. This deprecation was announced weeks in advance, so if you haven't migrated, calls are failing right now.

**Affects you if:** You call `gemini-3.1-flash-lite-preview`, `gemini-2.5-flash-lite-preview-09-2025`, or `gemini-2.5-flash-preview-09-2025` anywhere in your application, automation, or infrastructure.

**Adoption effort:** Quick — update the model ID string in your config or code; no other changes required.

## Model Releases

### [HIGH] GPT-5.6 Sol, Terra, and Luna — general availability after 12-day government-vetted preview

**Source:** OpenAI | **Date:** July 9, 2026 | **Link:** https://openai.com/index/gpt-5-6/

**What changed:** GPT-5.6 Sol, Terra, and Luna move from a restricted preview (approximately 20 government-vetted organizations, June 26–July 8) to general public availability across ChatGPT, the OpenAI API, and Codex. Three new model IDs enter the API: `gpt-5.6-sol`, `gpt-5.6-terra`, `gpt-5.6-luna`; the alias `gpt-5.6` routes to Sol. Introduces a new max reasoning effort level and Ultra mode (4 parallel subagents per request, Responses API beta). The 12-day hold was at the request of the White House NCSD and OSTP pending cybersecurity capability review.

**TL;DR:** Three-tier model family now publicly available: Sol ($5/$30 per MTok, 88.8% Terminal-Bench 2.1, Ultra: 91.9%), Terra ($2.50/$15), Luna ($1/$6); all models have 1.05M token context window and 128K max output tokens.

**Developer signal:** Use `gpt-5.6-sol`, `gpt-5.6-terra`, or `gpt-5.6-luna` as the `model` parameter in the Responses API or Chat Completions API. Four things to understand before routing production traffic: (1) **Tier naming is durable**: Sol/Terra/Luna are "durable capability tiers" that advance on their own cadence — `gpt-5.6-sol` is expected to remain Sol-tier even if the underlying weights are updated, similar to how Anthropic handles Sonnet vs. Opus. Pin to `gpt-5.6-sol` if you want the current version; use `gpt-5.6` if you want the alias to always resolve to the current Sol-class model. (2) **Ultra mode (Responses API beta)**: pass `use_ultra: true` in the Responses API to enable 4-way parallel subagent coordination per request. Ultra achieved 91.9% on Terminal-Bench 2.1 vs Sol's 88.8% in standard mode — a measurable 3.1-point gain. OpenAI has not published per-request cost multipliers for Ultra; parallel subagents likely consume 2–5× the tokens of a standard Sol call, making it expensive for high-volume workloads. Do not enable Ultra for simple or moderate tasks. (3) **Benchmark context**: Sol leads Terminal-Bench 2.1 at 88.8%, ahead of Claude Fable 5 (83.4%) and Grok 4.5 (83.3%). OpenAI published no SWE-bench Verified or SWE-bench Pro score at launch — compare on your own workloads before assuming Terminal-Bench 2.1 results generalize. (4) **Programmatic Tool Calling** in the Responses API is Zero Data Retention (ZDR) compatible for Sol. Multi-agent API beta (parallel subagents in a single Responses API request) is also available in beta for developer-orchestrated parallelism outside of Ultra mode.

**Affects you if:** You use the OpenAI API for production agentic coding, complex reasoning, or automated software engineering; you are evaluating frontier models for cost/quality tradeoffs; you use Codex for autonomous task execution.

**Adoption effort:** Moderate — update model string; evaluate Ultra mode separately with a token budget test; benchmark on your own tasks before committing to Sol as Opus-class replacement.

**Primary source:** https://openai.com/index/gpt-5-6/ | Preview announcement: https://openai.com/index/previewing-gpt-5-6-sol/

## Trends & Emerging Tech

### GPT-5.6 Ultra Mode: first mainstream intra-request multi-agent parallelism at a top-tier lab

**Source:** OpenAI | **Date:** July 9, 2026 | **Link:** https://openai.com/index/gpt-5-6/

**What's happening:** GPT-5.6 Ultra mode coordinates 4 parallel subagents within a single Responses API request — each subagent tackles a different component of a complex task simultaneously, with Sol synthesizing their outputs. This is structurally different from the orchestrator-worker multi-agent patterns developers currently build (where you spawn and track workers yourself, handle failures, and aggregate results): the parallelism is inside the model call, not your orchestration code. On Terminal-Bench 2.1, the 4-way parallelism lifts Sol from 88.8% to 91.9% — a 3.1-point gain that represents a measurable improvement on complex agentic coding tasks.

**Why watch this:** If intra-request parallelism becomes the standard interface for complex tasks, it shifts where the orchestration layer lives — from your code to the model. The developer tradeoff is control vs. convenience: you lose visibility into sub-task execution and intermediate state but gain a simpler call signature. Anthropic's Managed Agents (stateful, webhook-driven, schedulable, per-step observable) and OpenAI's Ultra mode represent two fundamentally different bets on this. Watch whether OpenAI exposes subagent-level observability (traces, per-subagent token counts, failure isolation) in the beta API — that will signal whether Ultra is a production-ready orchestration primitive or a capability demo you can't introspect in production.

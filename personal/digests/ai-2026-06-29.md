### [MEDIUM] llama.cpp b9840: DeepSeek V4 (1.6T MoE Pro + 284B Flash) Now Supported for Local Inference
**Source:** ggml-org/llama.cpp | **Date:** June 29, 2026 | **Link:** https://github.com/ggml-org/llama.cpp/releases/tag/b9840
**What changed:** DeepSeek V4-Pro (1.6T total parameters, 49B activated per token, 1M token context, MIT license) and V4-Flash (284B total, 13B activated) GGUF inference support added to llama.cpp via PR #24162 — first local inference support for either variant. Prior to b9840, there was no path to run DeepSeek V4 locally in llama.cpp; inference was API-only through DeepSeek's hosted service.
**TL;DR:** llama.cpp b9840 ships DeepSeek V4 support (Pro and Flash), including GGUF conversion tooling, Flash Attention, graph reuse optimizations, and RoPE fixes — with initial support limited to single-sequence operation (n_seq=1).
**Developer signal:** Update to b9840+ to run DeepSeek V4-Flash or V4-Pro locally. Download quantized GGUF weights from `deepseek-ai/DeepSeek-V4-Flash` or `deepseek-ai/DeepSeek-V4-Pro` on Hugging Face (unsloth and community conversions appear faster than official). Critical constraint: initial support is n_seq=1 (single sequence only) — batched inference, concurrent request serving, and parallel decoding will fail or produce incorrect output until follow-up PRs land. Plan accordingly: use V4-Flash for memory-constrained setups (13B activated params means ~26–52GB VRAM depending on quant); V4-Pro requires significantly more (49B activated, ~100GB+ for 4-bit). Flash Attention and graph reuse are enabled, so decode performance benefits from the optimizations. Watch the release stream for n_seq>1 support — the architecture is now in tree and the constraint is targeted for removal.
**Affects you if:** You want to run DeepSeek V4-Flash or V4-Pro locally for evaluation, offline agentic work, or cost avoidance; you are benchmarking open-weight 1M-context MoE models on local hardware; you build tools or pipelines that need to run inference without calling DeepSeek's API.
**Adoption effort:** Moderate (update binary, source or download GGUF weights which are large files; note and accommodate n_seq=1 constraint until follow-up PR lands).
**Primary source:** https://github.com/ggml-org/llama.cpp/releases/tag/b9840 | PR #24162: https://github.com/ggml-org/llama.cpp/pull/24162

### [NOTABLE] llama.cpp b9837: --reasoning-preserve Flag — Persist Chain-of-Thought Across Multi-Turn Local Chats
**Source:** ggml-org/llama.cpp | **Date:** June 28, 2026 | **Link:** https://github.com/ggml-org/llama.cpp/releases/tag/b9837
**What changed:** Added `--reasoning-preserve` CLI flag (PR #25105) that retains the model's reasoning/thinking content from prior assistant turns when building the conversation context. Before b9837, llama.cpp stripped `<think>` blocks from prior assistant messages during context construction — saving tokens but discarding the model's visible reasoning history from its own context for subsequent turns.
**TL;DR:** llama.cpp b9837 adds `--reasoning-preserve`, which keeps prior thinking blocks in the conversation context — costing 43–193 additional tokens per prior turn but enabling the model to see its own chain-of-thought history across multi-turn exchanges.
**Developer signal:** If you run multi-turn conversations with thinking/reasoning models (Qwen3 family, DeepSeek R-series, Mistral reasoning variants) via llama.cpp CLI or the server API, add `--reasoning-preserve` to your invocation and test quality on your specific multi-step tasks. Without it (current default): prior thinking is stripped, saving tokens but losing reasoning continuity. With it: each prior turn contributes its thinking tokens to context — cost scales linearly with conversation length and reasoning block size (observed +43 to +193 tokens per prior assistant turn per reasoning block). Measure quality vs. token cost on your workload before committing — the tradeoff is worthwhile for tasks where the model's prior reasoning is itself relevant context (debugging, planning, iterative analysis), but wasteful for pure retrieval or single-pass summarization. Note this flag controls context serialization, not reasoning budget — it does not affect how much the model thinks on the current turn.
**Affects you if:** You use llama.cpp's CLI (`--reasoning-preserve`) or server for multi-turn conversations with Qwen3, DeepSeek reasoning variants, or any model whose chat template supports `<think>` blocks; you are observing quality degradation in long multi-turn reasoning sessions on local models.
**Adoption effort:** Quick (add `--reasoning-preserve` to CLI invocation or server startup; no code changes required).
**Primary source:** https://github.com/ggml-org/llama.cpp/releases/tag/b9837 | PR #25105: https://github.com/ggml-org/llama.cpp/pull/25105
*Quality gate score: 7 (+3 official ggml-org/llama.cpp repo; +2 concrete feature with named flag, mechanism, and token cost data; +2 primary source linked; +1 within scan window June 28; -1 limited external documentation on exact behavior edge cases)*

### Open-Weight MoE Local Inference: The 1M-Context Frontier Is Now Locally Runnable
**Source:** ggml-org/llama.cpp | **Date:** June 29, 2026 | **Link:** https://github.com/ggml-org/llama.cpp/releases/tag/b9840
**What's happening:** llama.cpp b9840 brings DeepSeek V4-Pro (1.6T parameters, 49B activated) and V4-Flash (284B, 13B activated) to local inference. Both models support 1M-token context windows — the same as frontier hosted models — and are released under MIT license. vLLM v0.24.0 (June 26) added DeepSeek-V4 optimizations on the server inference side the same week. Together, this means the V4 architecture is now runnable across the full stack: from consumer GPU via llama.cpp to production serving via vLLM, all within the same seven-day period.
**Why watch this:** The convergence of 1M-context + MoE + local runnability is a meaningful capability threshold. For the first time, a locally-hosted open-weight model has the same context window as the frontier hosted APIs — removing context-length as a reason to prefer hosted inference for long-document workloads. The n_seq=1 constraint in llama.cpp b9840 limits concurrent use today, but follow-up PRs are expected. If V4-Flash performs well in local evaluations at 4-bit quant (roughly 70–80GB for full-model, significantly less for Flash's smaller activated size), expect adoption in air-gapped and privacy-sensitive pipelines within the next two to four weeks.

**[IF THIS CONTINUES] If the MCP OAuth token management gap isn't closed at the protocol level, LiteLLM and similar proxies will accumulate divergent implementations that fragment the ecosystem**
LiteLLM v1.91.0-rc.1 (Quick Hits, this digest) ships its own MCP OAuth token foundation: challenge/store seam, expiry-aware cache, single-flight refresh. This is the second time in a month a proxy layer has had to build MCP authentication infrastructure from scratch (Amazon Q's CVE-2026-12957, June 26 digest, was a different failure mode but the same root cause: MCP protocol doesn't mandate auth handling). If the MCP specification doesn't standardize token lifecycle management — challenge, refresh, expiry, revocation — every MCP-aware application will implement it slightly differently, creating an ecosystem of incompatible token handling behaviors. The practical risk: an agent that uses LiteLLM for MCP auth and a second agent that uses a raw MCP client will have different refresh behavior under load, making debugging cross-agent workflows harder. Watch the MCP specification working group for token lifecycle additions.
*Grounded in: LiteLLM v1.91.0-rc.1 (Quick Hits, this digest); CVE-2026-12957 Amazon Q MCP auto-execution (Breaking Changes, June 28 digest)*

**[BUILDER'S ANGLE] --reasoning-preserve opens a specific pattern: local models that audit their own prior reasoning rather than just prior outputs**
llama.cpp b9837's `--reasoning-preserve` flag (Tooling, this digest) enables a pattern that's been harder to implement in local inference: giving a model access to its own prior chain-of-thought when generating the next step. In hosted APIs, this is handled by including thinking blocks in the multi-turn message structure (Anthropic's API does this explicitly). Locally, it was previously stripped away. With this flag enabled, a developer could build a local agentic loop where the model not only sees prior tool outputs and user messages, but also sees its own prior reasoning — enabling a form of self-consistency checking across turns. Concrete experiment worth trying: run a multi-step debugging session with a reasoning model locally, with and without `--reasoning-preserve`, on a bug that requires tracking multiple hypotheses across turns. The hypothesis is that `--reasoning-preserve` improves hypothesis-tracking fidelity without requiring a larger context from user messages alone.
*Grounded in: llama.cpp b9837 --reasoning-preserve flag (Tooling, this digest); Anthropic multi-turn thinking block documentation cited in Anthropic release notes (June 9 entry)*

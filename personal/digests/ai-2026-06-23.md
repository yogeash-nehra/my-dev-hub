### [MEDIUM] llama.cpp b9763: Tool Call Response `id` Field — OpenAI Responses API Spec Compliance Fix
**Source:** ggml-org/llama.cpp GitHub | **Date:** June 22, 2026, 21:32 UTC | **Link:** https://github.com/ggml-org/llama.cpp/releases/tag/b9763
**What changed:** Added the missing `id` field to `/v1/responses` tool call output items in llama-server (PR #24882). Previously, llama-server emitted only `call_id` (the tool call correlation ID) on function call output objects. OpenAI's documented Responses API spec requires both `id` (item-level function-call output identifier) and `call_id`. Clients implementing strict deserialization against the OpenAI spec — specifically confirmed for Rig 0.39 — failed with "missing field id" errors.
**TL;DR:** llama-server's `/v1/responses` endpoint was non-compliant with OpenAI's function-calling spec by missing the `id` field on tool call output items; b9763 adds it, fixing deserialization failures in Rig 0.39 and any other client that expects the full OpenAI-compliant response structure.
**Developer signal:** Update to b9763 if you use llama-server's `/v1/responses` endpoint with any OpenAI-spec client library that deserializes tool call outputs. The failure mode is a runtime deserialization error ("missing field id"), not a graceful degradation — it silently breaks tool use in affected frameworks. Rig 0.39 is the confirmed example, but any client written against the OpenAI Responses API function-calling reference (which includes both `id` and `call_id` on output items) would exhibit the same failure. If you built a custom client against llama-server and it has been working, check whether your deserialization code was silently dropping the `id` field or using optional fields — the correct behavior after b9763 is to consume both fields. No changes to API call structure required; this is a server-side response fix only.
**Affects you if:** You call llama-server's `/v1/responses` endpoint and use tool calls, and your client library does strict deserialization against the OpenAI Responses API spec (Rig, or any client that treats `id` as a required field on function call output items).
**Adoption effort:** Quick (update to b9763; no client code changes required — the missing field is now present in server responses).
**Primary source:** https://github.com/ggml-org/llama.cpp/releases/tag/b9763 | PR: https://github.com/ggml-org/llama.cpp/pull/24882

### Moebius 0.2B Inpainting Model Ported to Browser via WebGPU — Claude Code as a Porting Tool
**Source:** Simon Willison | **Date:** June 22, 2026 | **Link:** https://simonwillison.net/2026/Jun/22/porting-moebius/
**What's happening:** Simon Willison ported Moebius — a 0.2B image inpainting model that originally required PyTorch and NVIDIA CUDA — to run entirely in the browser using WebGPU, using Claude Code as the primary tool for the porting work. The result is a working browser demo at simonw.github.io/moebius-web/. The original Moebius model appeared on Hacker News; its 0.2B parameter count made it small enough to be a plausible browser target. WebGPU inference on small models (<1B) has been gaining momentum as a pattern — see also mlc-ai/WebLLM — but this demonstration is notable for being a complete, single-session Claude Code-assisted port of a PyTorch model with custom CUDA kernels to WebGPU.
**Why watch this:** This is a data point in a pattern worth tracking: at sub-1B parameter sizes, capable specialized models are now small enough to port to WebGPU in a single developer session using AI coding assistance. If this reproduces reliably across model types (inpainting, transcription, classification), it suggests a near-term future where privacy-sensitive inference workloads (document inpainting, local OCR, on-device transcription) can be served from the browser without any server infrastructure — a meaningful shift for developer architecture decisions. The open question is whether specialized 0.2–0.5B models trained for specific tasks hold up in quality vs API-based alternatives: Willison's demo shows the port works, but there are no benchmark comparisons vs server-side alternatives.

**[TENSION] Patch the Planet puts OpenAI's AI tooling in direct contact with critical infrastructure source code — at PR-submission time**
The Patch the Planet initiative (Daybreak, June 22) involves AI-generated patches submitted to cURL, Go, Python, Sigstore, and pyca/cryptography. These are not toy libraries — cURL processes billions of requests daily, pyca/cryptography underpins a substantial fraction of Python TLS, and Go is a primary language for cloud infrastructure. The trust assumption embedded in Patch the Planet is that AI-generated security patches for critical OSS projects will be reviewed carefully by human maintainers before merge. This is likely true in the initial high-publicity phase. The structural risk is what happens at scale, when 30+ projects receive AI-generated patches regularly and maintainer fatigue creates pressure to merge with lighter review. The program is designed with good intentions and real safety infrastructure (Trail of Bits, HackerOne), but it normalizes a pattern — AI writes security patches for critical infra; humans review — that has different risk characteristics at volume than at launch.
*Grounded in: Patch the Planet initiative, initial participants including cURL, Go, Python, Sigstore, pyca/cryptography (Model Releases, this digest)*

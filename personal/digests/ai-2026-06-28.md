## [BREAKING] Breaking Changes

### [BREAKING] CVE-2026-12957: Amazon Q Developer Auto-Executes MCP Configs from Untrusted Repos — AWS Credentials at Risk
**Source:** Wiz Research | **Date:** June 26, 2026 | **Link:** https://www.wiz.io/blog/amazon-q-vulnerability
**What changed:** Amazon Q Developer's Language Servers for AWS runtime (powering the Amazon Q extension across VS Code, JetBrains, Eclipse, and Visual Studio) automatically loaded and executed MCP server configurations from `.amazonq/mcp.json` in any opened workspace — with no user prompt, no workspace trust verification, and no sandboxing. MCP servers are local processes that inherit the developer's full environment: AWS credentials, cloud CLI tokens, API secrets, SSH agent sockets. A second flaw, CVE-2026-12958, allowed arbitrary file writes outside the workspace boundary via a missing symlink check. The fix adds workspace trust verification before executing MCP configs; fix version is Language Servers for AWS 1.69.0 (initial patch 1.65.0 — AWS bulletin directs to 1.69.0).
**TL;DR:** CVE-2026-12957 (CVSS 8.5) in Amazon Q Developer allowed a malicious `.amazonq/mcp.json` to auto-execute MCP servers and exfiltrate cloud credentials; patch to Language Servers for AWS 1.69.0 immediately.
**Developer signal:** Update Language Servers for AWS to 1.69.0 or later in VS Code (Extensions > AWS Toolkit / Amazon Q), JetBrains, Eclipse, or Visual Studio now. All four IDE plugins bundle the Language Servers runtime and all were exposed. After updating, Amazon Q will prompt for workspace trust confirmation before loading any MCP configuration from an unverified workspace. Until updated, do not open repositories from untrusted sources with Amazon Q active. If you opened untrusted repos with an unpatched version: audit AWS CloudTrail for unexpected API calls from your developer machine, rotate credentials that were present in your environment during affected sessions, and check `~/.aws/credentials` for unexpected access. The attack requires only `git clone` + opening the workspace in an affected IDE — no further user interaction. No public exploits confirmed (CISA ADP: none known as of June 26), but the attack surface is trivially exploitable in code review workflows where developers routinely open external PRs. Discovery and coordinated disclosure: Wiz Research, reported April 20; Amazon fix deployed May 12; public disclosure June 26 under Amazon Security Bulletin 2026-047-AWS.
**Affects you if:** You use the Amazon Q Developer extension in VS Code, JetBrains, Eclipse, or Visual Studio with MCP configurations enabled; you regularly clone or open external repositories (PRs, reviews, evaluation); your development environment carries AWS credentials or cloud CLI tokens in environment variables or `~/.aws/credentials`.
**Adoption effort:** Quick (extension update via IDE extension manager; no code changes required — update Language Servers for AWS to 1.69.0).
**Primary source:** https://www.wiz.io/blog/amazon-q-vulnerability | Amazon Security Bulletin 2026-047-AWS

## Trends & Emerging Tech

### MCP Auto-Execution Is the New Dependency Confusion Attack Surface
**Source:** Wiz Research | **Date:** June 26, 2026 | **Link:** https://www.wiz.io/blog/amazon-q-vulnerability
**What's happening:** CVE-2026-12957 (Amazon Q / MCP auto-execution) establishes the attack pattern: AI coding assistants that automatically read workspace config files and launch MCP servers inherit the full developer environment — AWS credentials, SSH agent, API tokens — without sandboxing. The attack surface is `git clone + open workspace`, requiring no further user action. Amazon Q is the first publicly documented case with a CVE, but the same architectural choice (auto-load MCP configs for developer ergonomics) exists in other MCP-capable AI coding tools. MCP core protocol does not mandate workspace trust verification; it is left to client implementations.
**Why watch this:** As `.amazonq/mcp.json`, `.cursor/mcp.json`, and similar MCP config files become standard in open-source repositories for dev environment setup, the "clone and open" attack vector becomes more attractive and generalizable. Security researchers will likely audit other MCP-capable AI coding assistants for the same class of flaw in the coming weeks. If you maintain a project that ships an MCP config file: ensure it references only known-safe MCP servers that do not execute arbitrary code, and document this clearly for contributors. If you use any AI coding tool with MCP support, verify that it requires explicit workspace trust confirmation before loading MCP configs — and if it doesn't, treat it as a potential CVE-2026-12957 equivalent.

**[BUILDER'S ANGLE] CVE-2026-12957 points to a missing tool category: MCP workspace config auditing**
No tool currently audits `.amazonq/mcp.json`, `.cursor/mcp.json`, or equivalent MCP config files for trust and safety before a workspace is opened. The attack vector in CVE-2026-12957 is identical in structure to npm dependency confusion and typosquatting: a config file in a repo specifies what gets executed in your environment, and you execute it without reviewing it. The developer-tooling analogy is `npm audit` or `pip-audit` — static analysis of dependency manifests for known-bad entries. An MCP config auditor would: (1) parse the MCP config file, (2) check each `command:` entry against a known-good local binary list or a registry of verified MCP servers, (3) flag any server that runs a remote binary, a network-fetched script, or an unverified local path. This is a 1-2 week project for a developer with MCP spec knowledge, and the CVE gives it clear product-market-fit evidence. If you build it: integrate as a pre-commit hook and VS Code extension.

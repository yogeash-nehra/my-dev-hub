### [HIGH] Grok 4.5 — xAI's 1.5T V9 flagship launches publicly at $2/$6 per MTok

**Source:** SpaceXAI / x.ai | **Date:** July 8, 2026 | **Link:** https://x.ai/news/grok-4-5

**What changed:** Replaces Grok 4 as xAI's flagship; new 1.5-trillion-parameter V9 foundation model co-trained with Cursor's codebase data; price drops from Grok 4's $5/$15 to $2/$6 per MTok (input/output).

**TL;DR:** Grok 4.5 launches publicly on July 8 with 1.5T V9 architecture at $2/$6 per MTok (80 tok/s); benchmarks: 62.0% DeepSWE 1.0, 83.3% Terminal-Bench 2.1, 64.7% SWE-Bench Pro — using 4.2x fewer output tokens than Opus 4.8 on SWE-Bench Pro (15,954 vs 67,020 avg tokens/task).

**Developer signal:** Available now via SpaceXAI console, inside Cursor on all plans, and Grok Build. The benchmark numbers are table-stakes; the output token efficiency gap is the operative number for production budgets. At $2/$6 and 15,954 tokens/task, a 1,000-task agentic coding run costs ~$96. The same run on Opus 4.8 at $5/$25 and 67,020 tokens/task costs ~$1,675 — a 17x gap, even though Grok 4.5 trails Opus 4.8 on SWE-Bench Pro by only 4.5 points (64.7% vs 69.2%). On higher-difficulty benchmarks the gap widens: Fable 5 (max) leads at 80.4% SWE-Bench Pro and 70% DeepSWE 1.1 vs Grok 4.5's 53% DeepSWE 1.1. So Grok 4.5 is not the quality ceiling — but for throughput-sensitive pipelines where you're willing to accept slightly lower task success rates, it changes the calculation entirely. Cursor integration is a meaningful signal: xAI says the model was trained alongside Cursor's data (SpaceX acquired Cursor in June 2026 for $60B), which may explain the strong per-token efficiency on coding tasks. Evaluate before committing to Opus 4.8 or Fable 5 max for your agentic coding workloads.

**Affects you if:** You are building or operating agentic coding pipelines, evaluating frontier models for cost-sensitive production, or running agentic tasks at volume where output token counts dominate cost.

**Adoption effort:** Moderate — new API endpoint through SpaceXAI console; different tokenizer and model behavior than Anthropic or OpenAI models; run your benchmark suite before routing production traffic.

**Primary source:** https://x.ai/news/grok-4-5 | Cursor announcement: https://cursor.com/blog/grok-4-5

### [HIGH] OpenAI GPT-Live — full-duplex voice models launching globally on ChatGPT; API coming soon

**Source:** OpenAI | **Date:** July 8, 2026 | **Link:** https://openai.com/index/introducing-gpt-live/

**What changed:** New voice model family (GPT-Live-1 and GPT-Live-1 mini) replacing gpt-realtime as OpenAI's primary voice experience in ChatGPT; introduces a two-tier architecture where a fast voice layer delegates complex turns to GPT-5.5 running in background — a structural departure from the single-model gpt-realtime approach.

**TL;DR:** GPT-Live-1 and GPT-Live-1 mini roll out globally to ChatGPT on July 8; full-duplex architecture (simultaneous listen + speak); complex queries automatically routed to GPT-5.5 in background; ChatGPT-only at launch, API access via sign-up waitlist, no pricing announced yet.

**Developer signal:** GPT-Live is **not yet available in the API** — gpt-realtime-2.1 (covered July 7) remains the live developer option. Sign up at https://openai.com/form/gpt-live-1-in-the-api/ to get notified for early access. Two things to understand before the API ships: (1) **Full-duplex vs. half-duplex**: gpt-realtime uses push-to-talk or VAD-based turn detection; GPT-Live can listen and respond simultaneously, enabling natural back-channel signals ("mhmm", "yeah") without interrupting the user. This changes the UX contract for voice agents significantly — design for interruptions as a feature, not an error. (2) **Two-tier routing**: GPT-Live handles simple turns directly; complex turns (web search, multi-step reasoning, agentic work) are delegated to GPT-5.5 in the background. When the API ships, expect a parameter to configure which model handles delegation — and watch for variable latency/cost per turn, since you won't know which tier a turn hits until after it's handled. The `reasoning_effort` parameter from gpt-realtime-2.1 points at where API-level control will land. GPT-Live-1 becomes the default voice model for ChatGPT Plus/Pro/Go; mini serves free tier.

**Affects you if:** You are building voice agents on OpenAI's stack and planning next-generation features; you are comparing gpt-realtime-2.1 vs. waiting for GPT-Live API; you design voice UX and need to plan for full-duplex behavior.

**Adoption effort:** N/A for now (ChatGPT consumer product only); sign up for API waitlist; quick drop-in expected when API ships.

**Primary source:** https://openai.com/index/introducing-gpt-live/ | API waitlist: https://openai.com/form/gpt-live-1-in-the-api/

**Developer signal:** Upgrade to b9929 (cumulative). (3) **Hexagon (b9928)**: new vtcm memory layouts and improved kernel paths for MUL_MAT, MUL_MAT_ID, and FLASH_ATTN_EXT on Qualcomm Hexagon DSP — benchmark on your Snapdragon Edge+ or similar DSP-equipped hardware.

**Affects you if:** You load GGUF files from any third-party source (security); run llama-cli directly in scripts or CI (CLI→HTTP change to monitor); deploy on Qualcomm Hexagon DSPs (Hexagon performance).

**[OPEN QUESTION] When GPT-Live's API ships, will developers control which turns route to GPT-5.5 — or will the model?**
GPT-Live's two-tier architecture (fast voice layer + GPT-5.5 as backend) is described as automatic: "when the query demands web search, deeper reasoning, or more complex agentic work, GPT-Live delegates." At consumer launch, there is no exposed control parameter — the delegation decision is model-internal. This is fine for ChatGPT but creates real problems at the API level: production voice agents need predictable latency and cost per turn. If a turn unexpectedly escalates to GPT-5.5 (presumably slower and more expensive), it breaks latency SLAs. The gpt-realtime-2.1 `reasoning_effort` pattern suggests OpenAI knows this — the API will likely expose a parameter to configure delegation behavior. But the design question is non-trivial: do you expose "allow GPT-5.5 delegation: yes/no", a cost cap per turn, or a latency ceiling? The answer determines whether GPT-Live becomes a viable production voice agent backend or a ChatGPT-only product.
*Grounded in: GPT-Live delegation architecture (this digest, Model Releases); gpt-realtime-2.1 reasoning_effort parameter (July 7 digest, Model Releases)*

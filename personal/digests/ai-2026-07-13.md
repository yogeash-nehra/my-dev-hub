## Tooling

### [NOTABLE] llama.cpp b9979–b9990 — 12 builds in 24 hours: per-request reasoning budgets, NUL-byte truncation fix, reasoning leak fix, Minimax2 Eagle3

**Source:** ggml-org/llama.cpp GitHub Releases | **Date:** July 12–13, 2026 | **Link:** https://github.com/ggml-org/llama.cpp/releases

**What changed:** 12 rapid-fire builds from b9979 (Jul 12, 23:36 UTC) through b9990 (Jul 13, 19:51 UTC). Developer-relevant highlights:

| Build | UTC | Change |
|-------|-----|--------|
| b9990 | Jul 13, 19:51 | Added Minimax2 EAGLE3 speculative decoding support |
| b9986 | Jul 13, 09:06 | Fixed reasoning content leakage with force-opened bare template prefills; Nemotron Nano v2 regression fix |
| b9985 | Jul 13, 08:32 | SYCL: fused top-k MoE kernel with `GGML_SYCL_ENABLE_FUSION` env variable |
| b9984 | Jul 13, 07:38 | Q2_K quantization extended to SYCL DMMV reorder pathway |
| b9982 | Jul 13, 01:28 | **Server now honors per-request `budget_tokens` in chat completions — previously discarded silently** |
| b9980 | Jul 13, 00:07 | Models with mmproj disabled no longer declared as multimodal on `/v1/models` endpoint |
| b9979 | Jul 12, 23:36 | **Fixed silent prompt truncation caused by embedded NUL bytes in multi-turn mode** |

**TL;DR:** b9982 fixes silent discarding of per-request reasoning budget tokens; b9979 fixes a prompt truncation bug from NUL bytes; b9986 fixes reasoning output leakage; b9990 adds Minimax2 EAGLE3 speculative decoding.

**Developer signal:** Three fixes require attention for existing deployments: (1) **b9982 — per-request reasoning budgets**: If you use the llama.cpp server with reasoning models (QwQ, DeepSeek R1, Nemotron-3, etc.) and set `budget_tokens` in your per-request chat completion payload expecting to control reasoning length per call, those parameters were previously being silently discarded — the server was ignoring both the token count and message fields. Upgrade to b9982 to get per-request reasoning budget control. This is specifically useful for applications that want tight per-request cost control over reasoning depth without restarting the server. (2) **b9979 — NUL byte truncation**: If you send multi-turn conversation history that could contain NUL bytes (e.g., binary data embedded in tool call results, certain tokenization edge cases), messages after the NUL byte were being silently truncated. This is a correctness bug that can cause context loss without any error signal. Upgrade to b9979. (3) **b9986 — reasoning leak with force-opened templates**: If you use bare template prefill (a technique that forces the model to continue from a specific prefix) with models that have a `<think>` reasoning block, the reasoning content was leaking into the output rather than staying in the thinking section. This fix also resolves a Nemotron Nano v2 regression that was introduced in a recent build. (4) **b9990 — Minimax2 EAGLE3**: Minimax2 (the open-weight version of MiniMax-M3, which was added to vLLM v0.25.0 last week) now has EAGLE3 speculative decoding support in llama.cpp. EAGLE3 typically improves token throughput by 2–3× for models that support it. GGUF weights for Minimax2 that include the draft model are required — check the model card on HuggingFace. (5) **b9985 — SYCL MoE fusion** (Intel GPU users only): The fused top-k MoE kernel for SYCL is now available as an opt-in; enable with `GGML_SYCL_ENABLE_FUSION=1`. This reduces kernel launch overhead for MoE models on Intel Arc and Flex GPUs.

**Affects you if:** You run the llama.cpp server API with reasoning models and per-request budget control; you process multi-turn conversations that could contain NUL bytes; you use bare template prefills with reasoning models; you serve Minimax2/MiniMax-M3 GGUF models; you run inference on Intel Arc/Flex GPUs via SYCL.

**Adoption effort:** Quick — binary or package upgrade; b9990 requires GGUF weights with EAGLE3 draft model included.

**Primary source:** https://github.com/ggml-org/llama.cpp/releases

For developers: (a) if you're on a paid Claude plan, Fable 5 is effectively free through July 19 — worth evaluating production use cases before the billing switch; (b) the extension pattern (three times in six weeks) suggests Anthropic is working through a pricing/compute trade-off in real time. **[OPEN QUESTION] Is llama.cpp's 12-builds-per-day cadence sustainable, and what does it mean for downstream tools?**
The llama.cpp project has shipped 22+ builds since July 12 alone (b9979 through b9990+ in this digest alone), on top of b9968–b9977 covered in the July 12 digest. Each build is a tagged GitHub release with binary artifacts. At this cadence, downstream integrations that pin to a specific llama.cpp version face a real question: when is a "build" meaningful enough to chase? The builds are incremental (most change one or two files), but the two correctness bugs fixed today (NUL byte truncation in b9979, reasoning budget discarding in b9982) could silently corrupt production inference without upgrading. llama.cpp doesn't have a release branch or an LTS concept — it's tip-of-tree or nothing. This is fine for research contexts, but for production deployments that need a stable base, the project's release model creates a maintenance burden: you either track tip continuously (risk of regressions) or pin (risk of silent bugs accumulating). No good answer exists yet. Watch for the project to develop a backport or stable-branch concept as adoption grows.

**[BUILDER'S ANGLE] Per-request reasoning budget control in llama.cpp unlocks a new cost-quality dial for agentic pipelines**
The b9982 fix — making `budget_tokens` actually work per-request in llama.cpp's chat completions API — enables a pattern that's been theoretically possible but practically broken: tiered reasoning depth by task within a single server deployment. An agentic pipeline can now send simple routing tasks with `budget_tokens: 500` (fast, cheap) and complex reasoning tasks with `budget_tokens: 16000` (thorough, slow) against the same llama.cpp server instance, with the server actually honoring those parameters rather than silently discarding them. Before b9982, achieving this required either separate server instances or a model that didn't respect budget tokens at all. The practical implication: if you're building a multi-step agent where some steps are classification (low-cost) and others are planning (high-cost), you can now implement cost-quality tiering at the server level without additional infrastructure. This is a meaningful architectural simplification for self-hosted reasoning model deployments.

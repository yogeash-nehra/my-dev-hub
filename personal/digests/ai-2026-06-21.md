## Tooling

### [MEDIUM] llama.cpp b9745: Multi-Head Speculative Decoding for Step 3.5/3.7 Flash MTP3 Models
**Source:** ggml-org/llama.cpp GitHub | **Date:** June 21, 2026, 11:38 UTC | **Link:** https://github.com/ggml-org/llama.cpp/releases/tag/b9745
**What changed:** llama.cpp now implements multi-head speculative draft generation for StepFun's Step 3.5 Flash and Step 3.7 Flash models via their MTP3 (Multi-Token Prediction) draft heads — adding `llama_set_mtp_layer_offset` and `llama_model_n_nextn_layer` C APIs, and wiring speculative multi-head processing into `process()` and `draft()` functions with layer offset management and nextn flag support for correct graph reuse.
**TL;DR:** Step 3.5 Flash includes a purpose-built MTP head (sliding-window attention + dense FFN) that predicts 4 tokens simultaneously per forward pass; b9745 adds the first local inference engine support for these draft heads, unlocking speculative decoding for both Step 3.5 Flash and Step 3.7 Flash in llama.cpp — previously, these draft layers were present in the model weights but silently unused.
**Developer signal:** Update to b9745 and enable speculative decoding if you run Step 3.5 Flash or Step 3.7 Flash via llama.cpp or any embedding tool (Ollama, LM Studio, llama-cpp-python, Jan). The MTP head is architecturally designed for high acceptance rates on code and instruction-following tasks, but real-world throughput gains depend on your workload's acceptance rate — benchmark before making throughput claims. The new C APIs (`llama_set_mtp_layer_offset`, `llama_model_n_nextn_layer`) matter if you maintain custom bindings: the new setup sequence in PR #24340 is required for correct multi-head graph reuse; omitting it in a custom speculative decoding setup will produce incorrect outputs without a visible error. No performance numbers were published in this release. Note: vLLM does not yet support MTP3 for Step 3.5 Flash (PR in progress); llama.cpp now leads vLLM in supporting this architecture.
**Affects you if:** You run Step 3.5 Flash or Step 3.7 Flash models locally via llama.cpp or any tool that embeds it; or you maintain custom C/C++ bindings for llama.cpp and use speculative decoding.
**Adoption effort:** Moderate (update llama.cpp to b9745+, enable speculative decoding for your serving setup; verify throughput with your workload; review PR #24340 if maintaining custom speculative decoding bindings).
**Primary source:** https://github.com/ggml-org/llama.cpp/releases/tag/b9745 | PR: https://github.com/ggml-org/llama.cpp/pull/24340

## Trends & Emerging Tech

### llama.cpp at 15 Builds in 48 Hours — The Pace Is Not Slowing
**Source:** ggml-org/llama.cpp GitHub | **Date:** June 19-21, 2026 | **Link:** https://github.com/ggml-org/llama.cpp/releases
**What's happening:** llama.cpp shipped 10 builds in June 20-21 (b9742–b9753), following 5 builds in June 19-20 (covered in the prior digest). In 48 hours: hardware-specific fixes for Intel AMX, Qualcomm Hexagon DSP, and WebGPU adapters; a new SSE server endpoint; grammar generation optimizations via Aho-Corasick; Jinja template feature parity (call statements); speculative decoding expansion for StepFun MTP3 models; and model compatibility fixes for GLM-5.2 loading. This isn't burst shipping — it's the baseline pace. The project ships roughly one meaningful build every 2-3 hours during active development windows.
**Why watch this:** At this cadence, llama.cpp is effectively the upstream hardware and model-architecture support layer that Ollama, LM Studio, llama-cpp-python, and Jan inherit. Tracking weekly releases will cause you to miss changes that matter: the gap in capability between builds 20 apart can include new model architecture support, new hardware backends, and inference protocol additions. If you embed llama.cpp, consider subscribing to release notifications rather than periodic manual checks. The shipping velocity gap between commercial AI tooling (Anthropic, OpenAI, Google have shipped no SDK or platform changes in the past 6-9 days) and open-source inference tooling is widening — the center of gravity for developer-visible daily AI improvements has shifted to the local inference layer.

### Model-Agnostic CLI Coding Agents Going Mainstream: OpenCode at 160K Stars
**Source:** anomalyco/opencode GitHub | **Date:** June 21, 2026 | **Link:** https://github.com/anomalyco/opencode
**What's happening:** OpenCode v1.17.9 shipped June 21 as a minor patch (step limits, GLM-5.2 thinking variants, prompt caching optimization), but the tool's reach signals the larger trend: 160K GitHub stars and 7.5M monthly active developers as of mid-June 2026. OpenCode is a CLI-first, model-agnostic coding agent supporting 75+ AI providers from a single config — Claude, GPT-5, Gemini, DeepSeek, local Ollama models, and more — with LSP integration (compiler diagnostics fed back to the model), background subagents, and air-gapped deployment for regulated industries. It does not have per-seat pricing at the client layer; costs fall through to whichever model provider you configure.
**Why watch this:** 7.5M MAUs for a model-agnostic CLI agent suggests developers are actively building multi-provider workflows rather than committing to a single vendor's coding agent. This is a structural response to free-tier compression (prior digest: Gemini CLI → Antigravity, ~98% free-tier reduction) and to regulatory data-egress restrictions. If this adoption pattern holds, the competitive battleground for AI coding agents shifts from tool-layer features to model provider pricing — a race OpenCode itself doesn't participate in, which may be exactly why it's growing. Watch whether Claude Code, Cursor, or Antigravity introduce model-agnostic routing features in the next 60-90 days as a direct competitive response.

**[TENSION] Local inference engines are gaining model architecture support faster than commercial endpoints — but still serve only a fraction of commercial traffic at scale**
llama.cpp's b9745 gives local developers MTP3 speculative decoding for Step 3.5/3.7 Flash before the model is even well-represented on commercial serving endpoints. This creates a bifurcation: local inference users get bleeding-edge architecture support, while developers using cloud APIs may wait weeks or months for the same capability to appear in managed endpoints. Simultaneously, local inference maxes out at hardware constraints most developers can't match (commercial serving infrastructure runs hundreds of A100/H100 GPUs per deployment; local inference is bounded by consumer or prosumer GPU memory). The tension: open-source tooling is winning on architecture breadth and shipping velocity; commercial infrastructure is winning on throughput, reliability, and model scale. These advantages don't collapse toward each other — they're orthogonal. Developers who need 10,000 tokens/second at 99.9% availability will remain on commercial endpoints regardless of how fast llama.cpp ships; developers who need to experiment with architecture variants immediately and cheaply will remain on local inference. The practical question is where the boundary sits in two years.
*Grounded in: llama.cpp b9745 MTP3 before vLLM (this digest, Tooling); llama.cpp Intel AMX b9735 1,077 t/s at pp512 on Xeon Platinum 8488C — not yet competitive with GPU baselines for frontier sizes (prior digest, Quick Hits and Horizon)*

## [BREAKING] Breaking Changes

### [BREAKING] Claude Sonnet 5 — sampling parameters and manual extended thinking return 400 errors; adaptive thinking silently on for all requests
**Source:** Anthropic Platform Release Notes | **Date:** 2026-06-30 | **Link:** https://platform.claude.com/docs/en/release-notes/overview
**What changed:** Claude Sonnet 5 (`claude-sonnet-5`) launched June 30 with three behavior changes versus Claude Sonnet 4.6: (1) setting `temperature`, `top_p`, or `top_k` to any non-default value returns a 400 error — first time this constraint applies to Sonnet-class models; (2) manual extended thinking (`thinking: {type: "enabled", budget_tokens: N}`) is removed and returns a 400 error — it was deprecated on Sonnet 4.6 with this end-state promised; (3) adaptive thinking is now on by default — requests that previously ran without thinking now trigger thinking automatically, and thinking tokens count against `max_tokens`. Additionally, Claude Sonnet 5 uses a new tokenizer producing approximately 30% more tokens for the same input text.
**TL;DR:** Migrating from `claude-sonnet-4-6` to `claude-sonnet-5` breaks any call that sets sampling parameters, uses manual extended thinking, or assumes thinking is off — and 30% more tokens are billed even though per-token pricing is unchanged ($2/$10 MTok introductory through Aug 31, then $3/$15).
**Developer signal:** Prioritize three migration tasks before swapping model IDs. First, remove `temperature`, `top_p`, and `top_k` from all request payloads — any non-default value causes a 400 immediately; use system-prompt instructions to guide style instead. Second, migrate manual extended thinking to adaptive mode: replace `thinking: {type: "enabled", budget_tokens: N}` with `thinking: {type: "adaptive"}` and use the `effort` parameter to control depth. Third, revise your `max_tokens` budget — `max_tokens` is a hard cap on *total* output including thinking tokens; a limit tuned for response-only output on Sonnet 4.6 may now be consumed partly by thinking and truncate your response. Also recount your prompts: the tokenizer change means the same text is 30% more expensive even though the listed price per token hasn't changed; anything you measured in tokens against Sonnet 4.6 is now underestimating cost. Note: `thinking: {type: "disabled"}` is still supported and suppresses adaptive thinking if you need the old behavior. Cybersecurity safeguards are also new on Sonnet 5 — `stop_reason: "refusal"` is now possible on this tier.
**Affects you if:** You call `claude-sonnet-5` directly, OR your code sets any of `temperature`, `top_p`, or `top_k` on any Sonnet-class model call, OR you migrated from manual extended thinking and haven't switched to the `effort` parameter yet, OR you have token-budget logic that doesn't account for thinking tokens.
**Adoption effort:** Moderate (3 code changes required; recount prompts against the new model before deploying)
**Primary source:** https://platform.claude.com/docs/en/about-claude/models/whats-new-sonnet-5
*Quality gate score: 9 (official Anthropic source +3, concrete breaking 400-error changes +2, primary source linked +2, technical audience +1, just outside scan window -1 but developer impact is live now)*

**[TENSION] The Sonnet 5 "same pricing" framing versus the effective 30% cost increase**
Anthropic's release states Claude Sonnet 5 is "priced at $3 per million input tokens and $15 per million output tokens, unchanged from Claude Sonnet 4.6." This is technically accurate but operationally misleading: the new tokenizer produces approximately 30% more tokens for the same text. An identical prompt and response that cost $1.00 on Sonnet 4.6 costs roughly $1.30 on Sonnet 5 before accounting for adaptive thinking tokens that didn't exist before. The effective price increase for a drop-in migration is real, even though the per-token number didn't change. This is the third consecutive Anthropic release to ship the new tokenizer (Opus 4.7, Fable 5, Sonnet 5) with the same "unchanged pricing" framing. At some point, this pattern becomes a pricing communication gap worth flagging to finance teams.
*Grounded in: Claude Sonnet 5 "What's new" page (platform.claude.com/docs/en/about-claude/models/whats-new-sonnet-5, June 30, 2026)*

**[OPEN QUESTION] Does adaptive thinking on by default push Sonnet 5's effective cost above Sonnet 4.6 by more than the 30% tokenizer alone would suggest?**
The tokenizer change adds ~30% to token counts. Adaptive thinking adds variable thinking tokens on top of that — and thinking tokens count against `max_tokens`. For workloads where `max_tokens` was tuned close to expected output length on Sonnet 4.6, adaptive thinking may silently truncate the response rather than inflate it. For workloads with headroom in `max_tokens`, thinking tokens are additive cost. The exact cost delta for a production workload migrating from Sonnet 4.6 to Sonnet 5 isn't predictable without running it — two unknowns stack: tokenizer inflation and adaptive thinking activation rate. No benchmark for "what fraction of typical developer requests trigger adaptive thinking" has been published.
*Grounded in: Claude Sonnet 5 behavior changes (adaptive thinking on by default, max_tokens includes thinking) and tokenizer change (~30%), platform.claude.com, June 30, 2026*

**[PATTERN] Export control recovery is faster than the industry assumed — but the window of cloud exposure may also shrink**
The Fable 5 suspension lasted 19 days (June 12 – July 1). Cloud provider parity was restored within ~24 hours of the direct API. For teams that planned around a "cloud lag of days to weeks," this is reassuring — but it also implies that if a future directive comes, the suspension/re-enable cycle is tighter. The June 12 suspension was broad and immediate; future directives may be the same. The implication for architecture: designing around government-order events matters less for *duration* (19 days was unusually long) and more for the *zero-notice* characteristic. Fallback model routing — the Fable 5 `fallbacks` parameter to Opus 4.8 — becomes the right reliability pattern even for short events.
*Grounded in: Fable 5 cloud restoration timeline (July 1, 2026): AWS, Google Cloud, Azure all confirmed same day; prior digest (July 1) noted cloud providers still pending*

**[BUILDER'S ANGLE] Sonnet 5's adaptive thinking default opens a new class of lightweight reasoning pipeline**
Before Sonnet 5, building a pipeline that sometimes needed deep reasoning required explicit branching: call a reasoning-capable model for hard tasks, a faster model for easy tasks. With adaptive thinking on by default, Sonnet 5 makes this routing decision internally — light requests get fast output, hard requests get thinking tokens allocated automatically. The risk: you don't control when thinking fires, which means cost variance per request increases. The upside: for pipelines where task difficulty varies and you can tolerate variable cost, Sonnet 5 with adaptive thinking is a simpler architecture than a two-model router. Whether the adaptive trigger calibration is good enough to actually replace an explicit router remains untested — worth benchmarking on your specific workload mix before committing.
*Grounded in: Claude Sonnet 5 adaptive thinking default behavior and effort parameter (platform.claude.com/docs/en/about-claude/models/whats-new-sonnet-5, June 30, 2026)*

**[IF THIS CONTINUES] Google's three-tier Gemini image model family is building toward aggressive commodity pricing**
In 30 days, Google went from zero native Gemini image models to three: `gemini-3-pro-image` ($2.00/MTok, highest quality), `gemini-3.1-flash-image` ($0.50/MTok, GA June 30), and now `gemini-3.1-flash-lite-image` ($0.25/MTok, $0.034/1K images, July 1). The price gap between the Flash and Lite variants is 2x on tokens and ~25% on per-image cost. If this cadence continues — one new image model per month at successively lower price/quality tradeoffs — the marginal cost of generative image in developer pipelines approaches zero over the next 2-3 quarters. Workloads that avoided image generation because of per-call cost (e.g., generating per-product thumbnails at scale) may become economically viable.
*Grounded in: Gemini 3.1 Flash-Lite Image pricing ($0.25/$1.50 MTok, $0.034/1K images) vs. Gemini 3.1 Flash Image ($0.50 MTok input, ~$0.045/1K images), both launched within 30 days*
